Documentation Complète — Projet Ecommerce

Page 1/16 · Vue d’ensemble et objectifs

Cette documentation décrit en profondeur l’application Projet_ecommerce, sa construction technique, ses modules métier, et son fonctionnement côté client et administrateur. Le document est produit à partir de l’analyse directe du code source frontend + backend.

FrontendReact + TypeScript + Vite
BackendNode.js + Express + Zod
Base de donnéesMySQL / MariaDB
PaiementAirtel, Maisha Pay, WhatsApp, Livraison

But fonctionnel

  • Permettre la découverte de produits, la constitution d’un panier, la création de commandes et leur suivi.
  • Offrir un back-office administrateur pour gérer catalogue, catégories, commandes, transactions, utilisateurs, ventes et indicateurs.
  • Intégrer des canaux de paiement hybrides adaptés au contexte local (mobile money et assistance WhatsApp).

Périmètre couvert

  • Architecture technique, routage, sécurité (JWT + rôles), API REST, modèle de données.
  • Parcours client et admin détaillés, logique de stock, notifications, facturation PDF, chatbot.
  • Déploiement Railway, variables d’environnement, exploitation et recommandations production.

Page 2/16 · Architecture globale

L’application suit une architecture SPA + API avec séparation claire entre interface utilisateur et logique métier serveur.

Couche présentation (Client Web)

  • React RouterNavigation par routes protégées.
  • AuthContextStockage token + utilisateur.
  • Lib APIAppels HTTP avec timeout, headers et gestion d’erreurs.
  • UIHeader/Footer/Chatbot + pages métier.

Couche métier (API Express)

  • Validation des entrées avec Zod.
  • Contrôle d’accès par middleware JWT et rôles.
  • Transactions SQL pour commandes/paiements.
  • Génération de factures et rapports PDF avec Puppeteer.

Flux principal

Navigateur → /api/* (fetch) → Express → MySQL pool → réponse JSON/HTML/PDF

Points de fiabilité

  • Pool DB avec retries sur erreurs transitoires.
  • Health endpoints /api/health et /api/health/db.
  • Support Railway URL + variables individuelles.
  • Fallback SPA pour servir le build frontend.

Page 3/16 · Frontend: fondations techniques

Boot applicatif

  • main.tsx instancie React, BrowserRouter et AuthProvider.
  • App.tsx définit toutes les routes et protections d’accès.
  • Layout.tsx encapsule Header, Footer, Outlet, Chatbot.

Gestion de session

  • Utilisateur courant dans localStorage(auth_user).
  • AuthContext expose login/register/logout/refreshMe.

Client HTTP

Page 4/16 · Backend: structure et middlewares

Composants backend

  • server.js lance l’écoute HTTP.
  • index.js contient routes, services et middleware.
  • auth.js signature/validation JWT et contrôle de rôle.
  • db.js stratégie de connexion DB et résilience.

Sécurité et contrôle d’accès

  • JWT obligatoire pour les routes sensibles.
  • Roles stricts: client / admin.
  • Protection contre actions interdites (ex: suppression de son propre compte admin).

Middlewares et robustesse

  • Parser JSON avec gestion explicite des JSON invalides.
  • CORS multi-origines paramétrable via variable d’environnement.
  • Logger API pour traçabilité des requêtes.
  • SMTP vérifié au démarrage avec retry.

Services internes

  • Envoi facture (email + trace notification WhatsApp).
  • Services paiement (Maisha Pay / Airtel Money).
  • Génération PDF facture et rapport commandes.

Page 6/16 · Authentification, autorisation et sécurité

Inscription / connexion

  • Inscription: validation email + mot de passe, hash bcrypt côté API.
  • Connexion: vérification identifiants puis émission token JWT 7 jours.
  • Route /api/me: récupération profil courant.

Rôles et séparation des responsabilités

Client

  • Consulte produits/catégories/avis.
  • Gère panier local et adresses.
  • Passe commande et initie paiement.

Admin

  • Gère catalogue et catégories.
  • Valide paiements/livraisons.
  • Pilote ventes, utilisateurs, transactions et KPI.

Mesures de sécurité

  • Endpoints critiques protégés par double middleware auth + rôle.
  • Validation stricte des payloads par Zod.
  • Normalisation des URL d’images pour réduire les entrées invalides.
  • Aucun stockage de PIN Airtel durable côté front.

Page 7/16 · Parcours Client détaillé

1) Découverte et recherche produit

  • Filtres: catégorie, recherche textuelle, intervalle de prix, tri.
  • Affichage stock faible, images normalisées + fallback.

2) Panier

  • Stockage local cart_v1 (localStorage).
  • Actions: ajouter, modifier quantité, retirer, vider panier.
  • Calcul sous-total/total côté client.

3) Checkout et paiement

  • Modes: Maisha Pay, Airtel, WhatsApp, livraison.
  • Création commande + détails via API unique d’initiation.
  • Si Airtel: phase éventuelle de validation PIN.
  • Si WhatsApp: redirection vers conversation préremplie.

4) Post-achat

  • Consultation commandes et détails article par article.
  • Accès facture HTML/PDF et partage.
  • Ajout d’avis et gestion d’adresses.

Page 11/16 · Commandes, livraison, factures et ventes

Cycle de vie commande

  • en_attente → paye (validation paiement).
  • traitement → expedie → livre (logistique).
  • annule possible avec restauration de stock.

Règles critiques

  • Passage à payé déclenche création vente (idempotente).
  • Livré peut archiver/supprimer la commande active selon logique implémentée.
  • Échec paiement et annulation créent des notifications opérationnelles.

Facturation

  • Facture HTML consultable par référence commande.
  • Export PDF via Puppeteer (A4, impression propre).
  • QR code intégré pointant vers URL facture en ligne.

Rapport global PDF

  • Accessible admin, synthétise total commandes, CA, livraisons.
  • Liste détaillée des commandes pour audit.

Page 13/16 · Qualité logicielle et résilience

Validation et robustesse

  • Schémas Zod pour entrées API sensibles.
  • Gestion d’erreurs explicites sur toutes les routes majeures.
  • Retry SQL sur erreurs transitoires (ETIMEDOUT, ECONNRESET...).

Résilience réseau

  • Configuration DNS améliorée pour éviter les timeouts Node/Windows.
  • SMTP vérifié et reconnecté avec stratégie de tentatives.
  • Timeout frontend configurable pour fetch API.

Consistance des médias

  • Normalisation image centralisée front/back.
  • Support local upload converti pour portabilité en ligne.
  • Fallback visuel universel.

Limites actuelles identifiées

  • Panier majoritairement localStorage côté front (complété par API panier côté backend).
  • Certaines routes notifications (unread-count / mark-as-read) dépendent d’implémentation complète backend.
  • Multiplicité des modes paiement nécessite supervision opérationnelle stricte.

Service/api/chatbot, /api/factures/:reference, /api/admin/rapport-commandes/pdf

Convention de sécurité API

  • Public: consultation générale (produits, categories, avis, bannieres, health).
  • Client: panier, adresses, paiements, commandes personnelles.
  • Admin: gouvernance complète (stocks, utilisateurs, statuts, reporting).

Codes de statut usuels

  • 200/201: succès opérationnel.
  • 400: erreur de validation ou payload invalide.
  • 401/403: authentification ou droits insuffisants.
  • 404: ressource absente.
  • 500: erreur interne / intégration externe indisponible.

Page 16/16 · Exploitation, tests et feuille de route

Procédures d’exploitation

  • Démarrage local: frontend + API via scripts npm.
  • Surveillance: health endpoints, logs paiement, logs notifications.
  • Backup DB recommandé avant opérations massives admin.

Tests recommandés

  • Tests fonctionnels bout-en-bout: inscription → panier → paiement → facture.
  • Tests de rôle: blocages admin/client selon route.
  • Tests de non-régression sur stock et transitions commande.
  • Tests d’intégration Maisha/Airtel en sandbox.

Améliorations prioritaires

  • Unifier pleinement panier front et panier SQL backend.
  • Compléter endpoints notifications pour unread-count/mark-as-read.
  • Introduire pagination serveur sur gros volumes (produits, ventes, commandes).
  • Ajouter journal d’audit admin (qui a changé quoi et quand).

Conclusion

L’application est déjà structurée pour un usage réel avec une base technique solide. Les modules client et admin sont clairement séparés, les flux critiques sont transactionnels, et les canaux de paiement/communication sont adaptés à un contexte e-commerce moderne.
